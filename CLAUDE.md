# Personal AI Employee - Project Instructions

## Vault Location
The Obsidian vault is at `./AI_Employee_Vault/`

## Folder Structure
| Folder | Purpose |
|--------|---------|
| `Inbox/` | Drop zone for new files — monitored by the file system watcher |
| `Needs_Action/` | Items awaiting Claude processing (created by watchers) |
| `In_Progress/cloud/` | Files claimed by Cloud agent (Platinum — claim-by-move) |
| `In_Progress/local/` | Files claimed by Local agent (Platinum — claim-by-move) |
| `Updates/` | Cloud agent dashboard updates (Platinum — single-writer rule) |
| `Plans/` | Multi-step action plans generated by Claude |
| `Pending_Approval/` | Actions requiring human sign-off before execution |
| `Approved/` | Human-approved actions ready to execute |
| `Rejected/` | Actions the human declined |
| `Done/` | Completed and archived items |
| `Logs/` | Activity log files |
| `Briefings/` | CEO Briefing reports generated by `/ceo-briefing` skill |

## Work-Zone Specialization (Platinum)

The system supports two agent modes via `AGENT_MODE` environment variable:

### Cloud Agent (`AGENT_MODE=cloud`)
- Runs 24/7 on Oracle Cloud VM via Gemini free API
- **NEVER sends emails directly** — always uses `draft_email` (not `send_email`)
- **NEVER posts to social media directly** — only creates draft files in `Pending_Approval/`
- Creates approval files in `Pending_Approval/` for all actions
- Writes dashboard updates to `Updates/` (never edits `Dashboard.md` directly)
- Claims files via `In_Progress/cloud/`

### Local Agent (`AGENT_MODE=local`)
- Runs on developer's machine with full Claude Opus
- Has full execution permissions (send emails, publish posts, create invoices)
- Merges cloud `Updates/` into `Dashboard.md` (single-writer rule)
- Runs `ApprovedEmailSender` to auto-send approved email responses
- Claims files via `In_Progress/local/`

### Claim-by-Move Rule
When processing files from `Needs_Action/`:
1. Check if the other agent already claimed the file in `In_Progress/<other>/`
2. If not claimed, move the file to `In_Progress/<self>/` (atomic claim)
3. If already claimed by the other agent, skip it
This prevents double-work when both agents run simultaneously.

### Single-Writer Dashboard Rule
- Cloud agent writes updates to `Updates/*.md` files
- Only Local agent merges `Updates/` into `Dashboard.md`
- After merging, the update files are deleted
This prevents merge conflicts in git-synced vaults.

### Approved Email Sender (Local Only)
The `ApprovedEmailSender` watcher (Local agent only) monitors `Approved/` for files with `type: email_response` in frontmatter and automatically sends them via Gmail API. After sending, files are moved to `Done/`.

## How to Process Work

### 1. Read the Company Handbook First
Before processing any item, read `AI_Employee_Vault/Company_Handbook.md` for the rules of engagement. It defines:
- Communication guidelines and tone
- Financial thresholds requiring approval (anything over $100)
- Data privacy requirements
- Priority levels (High / Medium / Low)
- Escalation procedures

### 2. Process Items in Needs_Action
When asked to process the inbox (or when `claude_processor.py` triggers you):
1. List all `.md` files in `AI_Employee_Vault/Needs_Action/`
2. For each file, read its YAML frontmatter (`type`, `priority`, `status`)
3. Determine the correct action based on the file type and Company Handbook rules
4. Execute the action or create an approval request if the handbook requires it
5. Move the processed file to `Done/` when complete
6. Log what you did in `Logs/`

### 3. Update the Dashboard
After processing items, update `AI_Employee_Vault/Dashboard.md` with:
- Current file counts per folder
- Recent activity summary
- Any alerts or items needing human attention

### 4. Approval Workflow
If the Company Handbook says an action needs human approval:
1. Create a file in `Pending_Approval/` describing the proposed action
2. Do NOT execute the action
3. Note it on the Dashboard
4. The human will move it to `Approved/` or `Rejected/`

### 5. Logging
For every action taken, create or append to a daily log file in `Logs/` named `YYYY-MM-DD.md` with:
- Timestamp
- What file was processed
- What action was taken
- Result (success/failure/pending approval)

## Email Processing (Gmail Watcher)
The Gmail watcher monitors unread important emails and creates `EMAIL_<gmail_id>.md` files in `Needs_Action/`. These files have:
- `type: email` in frontmatter
- `from`, `subject`, `received`, `gmail_id` metadata
- `priority: high` (all important emails are high priority)

When processing `type: email` items:
1. Read the email content and assess urgency
2. Draft an appropriate response following Company Handbook communication guidelines
3. If the sender is a **new contact** (not seen before), create an approval request in `Pending_Approval/` before sending any reply
4. Log the action in `Logs/`

### Gmail Setup (User)
1. Enable Gmail API at https://console.cloud.google.com/
2. Create OAuth 2.0 credentials (Desktop app) and download `credentials.json` to project root
3. First run opens browser for Google authorization
4. Set `GMAIL_CREDENTIALS_PATH` env var if credentials are not at `./credentials.json`

## Plan Generation (Reasoning Loop)

The AI Employee follows a **Read → Think → Plan → Act** reasoning loop for complex tasks. When an item in `Needs_Action/` requires multiple steps to complete, Claude creates a structured Plan.md before taking action.

### When to Create a Plan
- **Always:** `type: task` items
- **Conditionally:** emails, WhatsApp messages, invoices, or requests that require 2+ distinct steps
- **Never:** simple file categorization (`type: file_drop`), LinkedIn post routing, or single-action acknowledgments

### Plan File Format
Plans are stored in `AI_Employee_Vault/Plans/` with the naming convention:
```
PLAN_<description>_<YYYYMMDD_HHMMSS>.md
```

Each plan contains:
- YAML frontmatter: `created`, `source_file`, `status`, `priority`
- **Objective** — what needs to be accomplished
- **Steps** — numbered checkboxes, with `**(REQUIRES APPROVAL)**` on gated steps
- **Approval Required** — list of steps needing human sign-off (per Company Handbook)
- **Resources** — links to relevant vault files

### Plan Lifecycle
1. Item arrives in `Needs_Action/` via a watcher
2. During processing, Claude assesses complexity
3. If multi-step: `/create-plan` generates a Plan.md in `Plans/`
4. Source file status is updated to `plan_created`
5. Plan steps are executed (with approval gates honored)
6. When all steps are complete, plan status is set to `completed`

### Using the Skill
Invoke `/create-plan` to generate a plan for a specific item. The skill reads the source file, consults the Company Handbook, and produces a structured plan with appropriate approval gates.

## Gmail MCP Server (Email Sending)

The Gmail MCP server (`mcp_servers/gmail_sender.py`) gives Claude tools to send emails and manage drafts directly via the Gmail API.

### Available MCP Tools

| Tool | Description | When to Use |
|------|-------------|-------------|
| `send_email` | Send an email (to, subject, body, optional cc/bcc/in_reply_to) | Replying to known contacts after drafting a response |
| `draft_email` | Create a Gmail draft without sending (to, subject, body) | When approval is needed before sending (new contacts, sensitive content) |
| `list_drafts` | List recent Gmail drafts (optional max_results) | To check existing drafts before creating duplicates |

### Approval Rules for Email Sending
- **Known contact:** Draft the reply, then call `send_email` directly
- **New/unknown contact:** Call `draft_email` to save a draft, create an approval file in `Pending_Approval/`, and do NOT call `send_email` until the human approves
- **Sensitive content** (financials, legal, personal data): Always create an approval file first

### Email Processing Workflow with MCP
1. Gmail watcher detects unread important email → creates `EMAIL_<id>.md` in `Needs_Action/`
2. Claude processes the email file, drafts an appropriate response
3. **If known contact:** call `send_email` MCP tool to send the reply directly
4. **If new contact:** call `draft_email` MCP tool to save a draft, then create an approval file in `Pending_Approval/`
5. Move processed file to `Done/`, log the action

### Setup
The MCP server is configured in `.mcp.json` and starts automatically when Claude Code launches. It shares OAuth2 credentials with the Gmail watcher (`credentials.json` + `token.json`). The token must have both `gmail.readonly` and `gmail.send` scopes — if upgrading from read-only, delete `token.json` and re-authenticate.


### Workflow
1. Use `/facebook-post` skill (or `/facebook-post "topic"`) to generate a draft
2. Draft is saved to `Pending_Approval/` for human review
3. Human moves file to `Approved/` to publish, or `Rejected/` to discard
4. The Facebook Poster background thread detects approved posts and publishes via Graph API
5. After publishing, the file moves to `Done/` and the action is logged

### When Processing Facebook Items in Needs_Action
- `facebook_comment`: Read the comment context, draft a reply. For known contacts use `reply_to_comment` MCP tool. For new contacts, create approval request.
- `facebook_message`: Assess urgency, draft response, follow new contact approval rules.
- `facebook_mention`: Assess context and draft engagement response if appropriate.

### Facebook Setup (User)
1. Go to https://developers.facebook.com/ and create a Meta App
2. Add **Facebook Login** and **Pages API** products to the app
3. In Graph API Explorer, generate a **Page Access Token** with these permissions:
   - `pages_show_list`
   - `pages_read_engagement`
   - `pages_manage_posts`
   - `pages_messaging`
   - `pages_read_user_content`
   - `read_insights`
4. Convert to a **long-lived token** (60 days) via the Access Token Debugger
5. Set environment variables:
   ```bash
   export FACEBOOK_ACCESS_TOKEN="your_page_access_token"
   export FACEBOOK_PAGE_ID="your_page_id"
   ```
6. The watcher and poster start automatically when the orchestrator detects the token

### Token Refresh
Page Access Tokens expire after ~60 days. To get a permanent token:
1. Get a long-lived User Token from the Access Token Debugger
2. Exchange it for a permanent Page Token via:
   ```
   GET /v21.0/{page-id}?fields=access_token&access_token={long-lived-user-token}
   ```
3. The returned Page Access Token never expires

## Ralph Wiggum Loop (Autonomous Task Completion)
For complex tasks (`type: task`), the system uses an autonomous re-execution loop:

1. `claude_processor.py` detects a complex task in `Needs_Action/`
2. Creates a state file in `.ralph_state/` tracking the task
3. Claude processes the task, tries to exit
4. Stop hook (`.claude/hooks/stop_hook.sh`) checks: is the source file in `Done/`?
5. **NO** → re-inject prompt, loop continues
6. **YES** → allow exit
7. Safety limit: max 10 iterations (configurable via `RALPH_MAX_ITERATIONS`)

The loop ensures multi-step tasks are fully completed before Claude moves on.

## Audit Logging
Every action is logged in two formats:
- **Markdown logs:** `Logs/YYYY-MM-DD.md` — human-readable activity log
- **JSON logs:** `Logs/YYYY-MM-DD.json` — structured JSONL for machine processing

JSON log entry format:
```json
{
  "timestamp": "2026-03-04T10:30:00+00:00",
  "action_type": "email_send",
  "actor": "claude_code",
  "target": "client@example.com",
  "parameters": {"subject": "Invoice #123"},
  "approval_status": "approved",
  "result": "success"
}
```

## Error Recovery
- All watchers use `@with_retry` decorator for exponential backoff on failures
- Orchestrator monitors thread health and auto-restarts dead threads (max 5 attempts)
- Browser-based watchers (WhatsApp, LinkedIn) handle crash recovery and reconnection
